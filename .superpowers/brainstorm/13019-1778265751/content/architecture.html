Klyra — Architecture Overview

Option B: per-monitor goroutines, leader election, ConfigMap state

ConfigMap: klyra-config
YAML — list of monitors (type, interval, thresholds) + action definitions
→
ConfigMap: klyra-state
JSON — alarm states + 24h event history, written by leader every ~10s
↓ loaded at startup
Engine (leader pod only)
Leader election via k8s Lease. Only one replica runs the engine at a time.

Loader — parses config, instantiates monitors + actions
Evaluator — reads results channel, applies threshold logic, updates alarm state
State writer — batches state to klyra-state ConfigMap every 10s, trims >24h entries
Dispatcher — fires action modules when alarm state changes
↓ spawns goroutines
Monitor Modules
Each monitor = one goroutine with its own ticker interval
Sends CheckResult{monitorID, status, value, ts} to results channel

k8s — pod/deploy/node/event watcher
http — status code, latency, body match
→
Action Modules
Triggered by Dispatcher on alarm state change (firing / resolved)

http — POST to URL with ntfy.sh headers:
Authorization: Bearer X-Priority X-Tags
↓ all replicas serve
HTTP Server (all replicas)
GET /api/status — current alarm state (read from klyra-state)
GET /api/history — 24h event timeline
GET /api/config — parsed monitor config (read-only view)
POST /api/silences — create silence / ack
GET / — serves React SPA
↓
React UI
Status dashboard 24h history timeline Config viewer Silence / ack controls
Polls /api/status every 30s (or configurable interval)
